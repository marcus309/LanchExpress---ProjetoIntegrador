# FASE 3 — 3.3 Plano de Comunicação

**Projeto:** LanchExpress — Aplicativo de pedidos para cantinas de escolas e universidades
**Disciplina:** PI III — Desenvolvimento de Sistemas Orientados a Web
**Período da fase:** Semanas 9 a 12 — Entrega: 27/04/2026

---

## 1. Objetivo do Plano de Comunicação

Garantir que todos os integrantes do grupo, o professor orientador e os
*stakeholders* (cantinas-piloto e usuários de teste) recebam, no formato e na
frequência adequados, as informações necessárias para acompanhar o andamento
do desenvolvimento, decisões técnicas, riscos e impedimentos.

## 2. Stakeholders e Papéis

| Stakeholder | Papel | Interesse na comunicação |
|---|---|---|
| Equipe de desenvolvimento (5–7 alunos) | Implementação | Status de tarefas, integrações, bugs |
| Professor orientador | Avaliador / mentor | Sprint Review, riscos e progresso |
| Cantina-piloto (Senac SP) | Cliente final | Demonstração funcional, prazos |
| Usuários de teste (alunos) | Validadores de UX | Versões para teste, formulário de feedback |

## 3. Canais e Frequência

| Canal | Uso | Frequência | Responsável |
|---|---|---|---|
| Reunião presencial / Discord (Daily) | Sincronização rápida do dia | Diária (15 min) | Scrum Master |
| WhatsApp do grupo | Avisos urgentes e bloqueios | Conforme necessidade | Todos |
| GitHub (Issues + Pull Requests) | Backlog, code review | Contínuo | Todos |
| Trello / GitHub Projects | Visualização do Kanban | Atualização diária | Scrum Master |
| Reunião de Sprint Review | Demonstração ao professor | Final de cada sprint | Equipe completa |
| Relatório de Status (este doc.) | Registro formal de progresso | Quinzenal | PO / Líder |

## 4. Matriz de Comunicação (RACI resumido)

| Atividade | PO | Scrum Master | Devs | Professor |
|---|---|---|---|---|
| Definir backlog | R | C | C | I |
| Priorizar Sprint | R | A | C | I |
| Implementar tarefas | I | I | R | I |
| Revisão de PR | I | A | R | I |
| Sprint Review | A | R | R | C |
| Avaliação final | I | I | R | A |

R = Responsável | A = Aprovador | C = Consultado | I = Informado

## 5. Política de Escalonamento

1. Bloqueio detectado em até **24h** → reportar no canal do grupo.
2. Bloqueio não resolvido em **48h** → escalonar ao Scrum Master e replanejar
   na próxima Daily.
3. Bloqueio que ameace a entrega da fase → reunião extraordinária com o
   professor orientador.
